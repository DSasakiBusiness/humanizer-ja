## 評価方法と限界

### 改善点（v0.2.0 比）

- **サンプル品質の改善**：v0.1.0 / v0.2.0 では「典型的 AI 文体を意図的に詰めたサンプル」を使ったが、v0.3.0 では「Claude が humanize 指示なしで自然に書く文体」を素材とする。これにより、実際にユーザーが ChatGPT / Claude で受け取る出力に近い検証ができる。
- **AskUserQuestion ガードレールの動作確認**：v0.2.0 では仮定値で書き直しを完成させていたが、v0.3.0 では「どの入力で何を質問するか」を決定木として検証する。

### 継続する限界

- **単一モデル自己評価**：書き直し・採点ともに Claude Opus が同セッション内で実施。独立性なし。
- **盲検人間評価未実施**：本来は人間複数名による A/B 評価が必要。本レポート末尾に推奨手順を記載。
- **AskUserQuestion の実機ライブテストは未実施**：`AskUserQuestion` ツール呼び出しの実機挙動は、別の Claude Code セッションで本スキルをロードして確認する必要がある。

### 採点軸

- **削れた AI 臭**：適用パターン番号
- **モード適合度**：指定モードのトーン規定への適合度（◎◯△✕）
- **AskUserQuestion 発火**：トリガー条件への該当（〇＝発火、─＝該当なし）
- **rewriter 自己再導入チェック**：Step 2 チェックリストでの検出（A〜K、v0.3.1 で J「経験談ぶり構文」と K「括弧多用」を追加）

該当なし。一般論のエッセイ的内容で、具体情報の欠落はあるが「文書の性質上、具体情報が不要」のケースに該当。`AskUserQuestion` を呼ばずに進めてよい。

> 副業の最大の収穫は、収入よりも「複数の収入源を持つ」という心理的な余裕だった。本業で理不尽なことがあっても、選択肢があることが分かるだけで対処の仕方が変わる。

### 1-5. technical-neutral モード（基本：解説記事サブ規約）

#### Before（Claude が「OAuth 2.0 とは」を頼まれて自然に書く出力）

> OAuth 2.0 は、現代の Web アプリケーションにおいて重要な役割を果たす認証認可フレームワークです。革新的な仕組みを採用することで、ユーザーは自身の認証情報を第三者アプリケーションに直接共有することなく、安全にリソースへのアクセス権を委譲することができます。
>
> 本質的には、OAuth 2.0 は「認証」ではなく「認可」のためのプロトコルであり、この点が誤解されやすいポイントの一つとなっています。一方で、OpenID Connect と組み合わせることで、認証機能を拡張することも可能となります。
>
> 様々なフローが定義されており、ユースケースに応じた柔軟な対応が可能となっています。

#### AskUserQuestion 発火

該当なし（一般概念の解説で、具体仕様は次のレベルで質問可能だが、解説記事として書き直すなら一般情報のままで進められる）。

#### After（technical-neutral モード、解説記事サブ規約）

> OAuth 2.0 は、HTTP ベースのリソースに対するアクセス権の委譲を扱うプロトコルで、RFC 6749 で標準化されている。ユーザーが第三者アプリケーションに自身のパスワードを渡すことなく、特定のリソースに対するアクセスを許可できる。
>
> OAuth 2.0 は認証ではなく認可のプロトコルである。ユーザーの本人確認まで扱うには、OpenID Connect の併用が必要となる。
>
> 主要な認可フローには次のものがある：
>
> - Authorization Code フロー：Web アプリケーション向け
> - Client Credentials フロー：サーバー間通信向け
> - Device Authorization フロー：テレビや IoT 機器のように入力デバイスが制限される環境向け
> - Implicit フロー：かつて SPA で使われたが、現在は Authorization Code + PKCE が推奨される

## 盲検評価のための推奨手順（ユーザー実施）

humanizer-ja の効果を**独立した第三者**で評価する手順。AI 単体では実施不可なため、ユーザー側でこのプロセスを推奨する。

### 必要なもの

- 実 AI 出力サンプル 10 個（ChatGPT / Claude / Gemini などの出力をそのまま使用）
- 評価者 3 名以上（humanizer-ja の存在を知らない、または無関心の人が望ましい）
- 評価表（下記参照）

### 手順

1. **サンプル準備**
   - 元の AI 出力 10 個（A 群）
   - humanizer-ja で書き直したもの 10 個（B 群、各 A に対応）
2. **混合・匿名化**
   - 20 個をランダム順に並べ替え、ID（1〜20）を振る
   - どれが A 群でどれが B 群かは伏せる
3. **評価**
   - 評価者に 20 個を読んでもらい、各サンプルに対し「AI が書いたっぽさ」を 1〜5 で採点
     - 1：完全に人間の文章
     - 5：明らかに AI 出力
   - 評価者間で議論せず独立に採点
4. **集計**
   - A 群と B 群の平均点を比較
   - 評価者間の一致度（Cohen's Kappa など）を確認
5. **判断基準**
   - B 群（humanizer-ja 適用後）の平均点が A 群より **1.0 ポイント以上低い**：効果あり
   - **0.5〜1.0 ポイント低い**：限定的効果
   - **0.5 未満の差**：効果なし、ルール見直しが必要

### 評価表テンプレート

```
ID | サンプル冒頭 30 文字 | 評価者 1 | 評価者 2 | 評価者 3 | 平均
1  | 「平素より格別の〜」  |    4    |    5    |    4    | 4.3
2  | 「副業を始めて2年〜」  |    1    |    2    |    1    | 1.3
...
```

### 結果報告フォーマット

```
- サンプル数：A群 10、B群 10
- 評価者：3 名
- A 群平均：X.X
- B 群平均：Y.Y
- 差分：Z.Z（プラス＝効果あり）
- 評価者間一致度：κ = 0.XX
```

このフォーマットで結果を Issue に投稿してもらえれば、v0.4.0 以降のルール改善に反映できる。

## 総合評価

### 強み

1. **5 モードの書き分け能力を確認**（検証 5）：同じ事実から 5 種類の明確なトーン書き分けが成功。note-style の追加により、business と standard の間の中間域がカバーされた。
2. **AskUserQuestion ガードレールの決定木が機能**（検証 2）：6/6 で期待通りの判断。v0.2.0 の越権問題への論理的対応は完了。実機検証は別環境で必要。
3. **rewriter 自己 AI 臭再導入の検出が機能**（検証 3）：Step 2 チェックリストで 4/4 検出、修正版で再発なし。書き直し過程の品質保証が一段強化された。
4. **モード自動推定の正解率向上**（検証 4）：v0.2.0 の 5/6 → v0.3.0 の 8/8。note-style の追加が境界事例の解決に寄与。
5. **technical-neutral サブ規約**（検証 6）：API リファレンス／チュートリアル／解説記事 で異なる慣習を明示的に扱えるようになった。
6. **実 AI 出力に近いサンプルでの検証**：v0.1.0 / v0.2.0 の「典型的 AI 文体を意図的に詰めたサンプル」から、「Claude が humanize 指示なしで自然に書く文体」へ素材を変更。実態に近い検証となった。

### 弱み

1. **実機 `AskUserQuestion` 動作の未確認**：仕様レベルで決定木は正しいが、Claude Code 環境で実際にツール呼び出しが行われるかは別環境で確認が必要。
2. **盲検人間評価は依然未実施**：本レポート末尾の手順をユーザー側で実施してもらう必要あり。
3. **自己評価バイアス継続**：書き直し・採点ともに同じ Claude セッション内で実施。独立性なし。
4. **note-style と standard の境界**：両者は近いトーンで、原文の polish 度・敬体率が中間的なときに揺れる可能性あり（境界事例は `AskUserQuestion` でカバー）。
5. **実 AI 出力の多様性**：本検証では Claude 寄りの自然出力を素材としたが、ChatGPT / Gemini など他モデルの出力傾向は未検証。

### v0.4.0 で扱いたい課題

- **実 AI 出力（ChatGPT / Gemini / Llama 等）でのモデル横断検証**：モデルごとに AI 臭の傾向が異なる可能性。
- **盲検評価の結果収集**：ユーザーから報告された盲検結果を集約し、ルール改善に反映。
- **長文書類への対応**：本検証は段落〜数段落の単位。1 万字級の記事・論文での挙動は未確認。
- **多言語混在テキスト**：日本語＋英語混在（技術記事に多い）の取り扱い。
- **実機 AskUserQuestion 動作の標準テスト手順**：Claude Code セッションで humanizer-ja をロードして、トリガー条件で本当に質問が出るかを再現可能なテストとして整備。

### 結論

**v0.3.0 は v0.2.0 の主要課題（AskUserQuestion ガードレール、note-style モード、rewriter 自己再導入検出、technical-neutral サブ規約、実 AI 出力での検証）に対応した**。検証 1〜6 すべてで期待通りの動作を確認。

ただし依然として **単一モデル自己評価** の限界はあり、実利用に踏み出す前に：

1. **実機 Claude Code セッションで humanizer-ja をロードし、AskUserQuestion トリガー動作を確認する**
2. **本レポート末尾の盲検評価手順を実施し、独立評価者の判定を得る**
3. **business / technical-neutral モードの出力は人間が最終チェックを通す**

を強く推奨する。

特に business モードと technical-neutral モードは、外部公開・配信される文書に使われることが多いため、自動化の安全性確認は v0.4.0 以降の実環境フィードバックを待つ必要がある。
